Real-Time Streaming Ingest

Stand up a Fabric Eventhouse, define KQL tables for three live data streams (hospital vitals, hospital movement, Metra trains), wire each Event Hub through an Eventstream, and watch live data start flowing.

⏱ 75 minutes
🎯 Goal: 3 streams → 1 Eventhouse
🛠 Eventhouse · Eventstream · KQL

Learning Objectives

  • Create your Eventhouse and KQL Database from scratch.
  • Create three KQL tables with explicit schemas + JSON ingestion mappings.
  • Build three Eventstreams (one per source) routing Event Hub → KQL.
  • Verify live ingestion with KQL queries on each table.
  • Understand when to use Eventstream's no-code path vs. KQL update policies.

  3. Enable OneLake availability

    By default, KQL data lives in the Eventhouse's hot storage tier only. Turning on OneLake availability mirrors every KQL table out as Delta in OneLake automatically - no copy job, no pipeline. That's what lets your Direct Lake semantic model (Module 5), Fabric Ontology (Module 6), and Data Agents (Modules 7-8) read the same streaming data without bespoke ETL.

    Inside the KQL Database, click the database name in the left tree → Database details (or the gear icon at the top). Find the OneLake availability tile and toggle it On. Confirm in the dialog.

    📷
    SCREENSHOT NEEDED
    module-3/02-onelake-availability.png
    KQL Database details pane with the OneLake availability toggle switched On
    ✓
    Do this before creating tables. When OneLake availability is on at the database level, every table you create afterwards inherits it for free. If you flip the toggle later, only tables created from that point forward are mirrored - existing tables need to be migrated manually.
    ⓘ
    One stream, two surfaces. After this toggle, every record Eventstream writes to a KQL table will also land as a Delta row in OneLake within ~1 minute. Hot path stays sub-second for the Real-Time Dashboard; warm path is queryable from Lakehouse SQL endpoint, Power BI Direct Lake, or any other OneLake consumer.

  4. Open a query window

    Click the Explore your data button (or the database name in the left tree → Query). You'll write all the KQL in this query editor.

Step 2 - Create All Three KQL Tables KQL

Run the following KQL block in the query editor. It creates all four tables, enables streaming ingestion, and registers JSON ingestion mappings (which Eventstream will reference).

ⓘ
Why explicit mappings? Eventstream's no-code path can also infer schema, but for production-grade ingest you want named mappings so schema drift in the source doesn't silently break ingestion. This is the same pattern the hackathon repo uses - and it's the ISV-grade way.

2a - Hospital Vitals + Hospital Movement

KQL 
// =========================================================
// HOSPITAL VITALS - patient telemetry
// =========================================================
.create table HospitalVitals (
    patient_id: string,
    age: int,
    gender: string,
    diagnosis_code: string,
    condition: string,
    heart_rate: int,
    bp_systolic: int,
    bp_diastolic: int,
    temperature_f: real,
    spo2: int,
    respiratory_rate: int,
    timestamp: datetime
)

.alter table HospitalVitals policy streamingingestion enable

.create table HospitalVitals ingestion json mapping 'HospitalVitals_mapping'

Step 4 - Verify End-to-End with KQL

Switch back to the KQL query editor and run these one at a time. Each should return ten recent rows with timestamps from the last minute or so.

KQL · validation 
HospitalVitals       | take 10
HospitalMovement     | take 10
TrainTelemetry       | take 10

// Row counts in last 5 minutes - all three should have > 0
union withsource=table HospitalVitals, HospitalMovement, TrainTelemetry
| where timestamp > ago(5m)
| summarize events = count() by table
| order by events desc

Bonus - When to Use KQL Update Policies

Eventstream gives you a no-code, JSON-passthrough path. But sometimes you want to transform data on the way in - e.g., flag critical patients at write time. KQL's update policy is how you do that.

Optional - Add a "vitals_alerts" derived table (skip if short on time)
KQL · update policy 
// 1. The destination - only critical readings end up here
.create table VitalsAlerts (
    patient_id: string,
    heart_rate: int,
    spo2: int,
    temperature_f: real,
    severity: string,
    timestamp: datetime
)

// 2. The function that filters + transforms
.create function VitalsAlerts_TransformFn() {
    HospitalVitals
    | where heart_rate > 120 or heart_rate < 50
         or spo2 < 90
         or temperature_f > 101.5
    | extend severity = case(
        spo2 < 88 or heart_rate > 140, "critical",
        "warning")
    | project patient_id, heart_rate, spo2, temperature_f, severity, timestamp
}

// 3. Bind the function to the destination - runs on every batch ingest
.alter table VitalsAlerts policy update
@'[{"Source":"HospitalVitals","Query":"VitalsAlerts_TransformFn()","IsEnabled":true,"PropagateIngestionProperties":true}]'
✓
Update policies run synchronously on the ingest path, so the derived table is always in lock-step with the source. Combine that with a Real-Time Dashboard tile (Module 4) for a true sub-minute alerting pipeline.
