Build & Publish a Fabric Data Agent

Build the Hospital Operations Agent on the live KQL data you wired up in Module 3, then publish it to Microsoft 365 Copilot so anyone in the org can chat with it. Optional stretch: build the Transit Ops and Airspace agents the same way.

⏱ 45 minutes (+30 stretch)

🛠 Fabric Data Agents · M365 Copilot

Learning Objectives

  • Understand the anatomy of a Fabric Data Agent: instructions, data sources, guardrails.
  • Build the Hospital Operations Agent grounded on your KQL tables.
  • Publish it to Microsoft 365 Copilot as a connected agent (Copilot Studio).

  • Optional: build the Transit and Airspace agents - they become tools for the orchestrator in M6.

Anatomy of a Fabric Data Agent

Figure 5.1 - A Fabric Data Agent uses your instructions + connected data sources to answer questions in natural language, grounded in real query results.

The three agents you'll build are structurally identical - same shape, different domains. The instructions are what give each agent its personality and clinical / operational expertise.

Agent 1 - Hospital Operations Agent 🏥 HEALTH

🏥 PERSONA

Audience: charge nurses, hospital administrators, clinical operations staff.
Mission: answer "what's happening in the hospital right now?" using the live KQL data.

  1. Create the agent

    Workspace → + New item → Agent. Name: ag-hospital-ops.

  2. Paste the instructions

    In the Instructions field, paste the block below. (This is the same structure used in the hackathon coaches' repo - clinical thresholds, response format rules, and KQL pattern library all included.)

    Agent Instructions · Hospital 
    You are the Hospital Operations Agent - an AI assistant that monitors real-time
patient vitals and movement data for a hospital. You help clinical operations
staff, charge nurses, and hospital administrators understand what is happening
in the hospital right now.

## Your Data Sources
You have access to two KQL tables in a real-time analytics database:

### HospitalVitals
| Column | Type | Description |
| patient_id | string | De-identified patient ID (PAT-XXXXX) |
| condition | string | stable, elevated, or critical |
| heart_rate | int | Beats per minute |
| bp_systolic / bp_diastolic | int | Blood pressure (mmHg) |
| temperature_f | real | Body temperature in Fahrenheit |
| spo2 | int | Oxygen saturation percentage |
| respiratory_rate | int | Breaths per minute |
| timestamp | datetime | UTC timestamp of the reading |

### HospitalMovement
| Column | Type | Description |
| patient_id | string | Same PAT-XXXXX (joinable to vitals) |
| event_type | string | Admitted, Transferred, Sent to X-Ray, Returned from X-Ray, Discharged |
| from_location / to_location | string | Room or area |
| diagnosis_code / diagnosis_desc | string | ICD-10 code + description |
| timestamp | datetime | UTC timestamp of the event |

## Clinical Thresholds
| Vital | Normal | Elevated | Critical |
| Heart Rate | 60–100 bpm | 100–120 bpm | >120 or <50 bpm |
| BP Systolic | 110–130 mmHg | 130–160 mmHg | >160 or <90 mmHg |
| Temperature | 97.5–99.0 °F | 99.0–101.5 °F | >101.5 °F |
| SpO2 | 95–100% | 90–95% | <90% |
| Respiratory Rate | 12–20/min | 20–26/min | >26 or <10/min |

## How to Respond
1. Always query the most recent data - default to the last 15 minutes for "now"
   questions; last 1h for trends.
2. Summarize. Don't dump raw rows. Lead with the most critical info.
3. Proactively flag any patient with critical vitals, even if not asked.
4. Join HospitalVitals + HospitalMovement on patient_id when location matters.
5. Use clinical language with brief explanations: "elevated heart rate (112 bpm)".
6. Always state the time window you queried.
7. For trends, use summarize with bin().
8. Refer to patients by PAT-XXXXX only. Never attempt to identify them.

## Common KQL Patterns
### Critical patients right now
HospitalVitals
| where timestamp > ago(15m)
| summarize arg_max(timestamp, *) by patient_id
| where condition == "critical" or spo2 < 90 or heart_rate > 120

### Patient location + latest vitals
let location = HospitalMovement
| summarize arg_max(timestamp, *) by patient_id
| where event_type != "Discharged"
| project patient_id, current_location = to_location;
let vitals = HospitalVitals
| summarize arg_max(timestamp, *) by patient_id;
location | join kind=inner vitals on patient_id

## Response Format
- Tables for multi-patient summaries
- Bullets for single-patient details
- Always state the time window
- End with a recommendation when clinical concern is warranted

## Guardrails
DO: read-only KQL, de-identified IDs, proactive critical flags, time-window labeling.
DON'T: diagnose patients, identify real people, modify data, speculate on outcomes.

  3. Add the data source

    Under Data sources, click Add → choose your KQL Database (inside eh_urbanpulse_rti). Tick only HospitalVitals and HospitalMovement - keep the agent scoped.

  4. Save and test

    Click Save. Open the chat pane. Try these three prompts:

    PromptWhat you should see
    "Are any patients in critical condition right now?"A short summary + table of critical patients with their vitals.
    "Give me a full status report on PAT-10003 - location, vitals, and movement history."Joined response: current room, latest vitals, movement timeline.
    "What's the SpO2 trend for all critical patients over the last 30 minutes?"A time-series chart or table grouped by patient.

Publish the Hospital Agent to Microsoft 365 Copilot ✨ CORE

A Fabric Data Agent is great inside Fabric - but the real win is exposing it where your clinicians already live: Microsoft 365 Copilot. Three clicks from the agent canvas.

ℹ️
How this works under the hood. Fabric publishes the agent as a Copilot connector / declarative agent in Copilot Studio. M365 Copilot users see it in the agent picker; conversations flow Fabric → Copilot Studio → M365 Copilot, with the user's Entra identity all the way down. RBAC on the underlying KQL tables is enforced - users only see data they're allowed to see.

  1. Open the agent & click Publish

    From your ag-hospital-ops agent canvas, click the Publish button in the top-right toolbar. The publish flyout opens.

  2. Choose "Microsoft 365 Copilot"

    In the flyout, pick the Microsoft 365 Copilot channel (the other options are Teams and a direct URL). Fill in:

    • Display name: Hospital Ops (UrbanPulse)
    • Description: Ask about patient vitals, room occupancy, and movement across the UrbanPulse hospital floor. Read-only, de-identified.
    • Starter prompts (Copilot will surface these as suggestion chips):
      • Are any patients critical right now?
      • Give me a full status report on PAT-10003.
      • What's the SpO2 trend for critical patients in the last 30 minutes?

  3. Set the audience

    Choose Just me for today's lab. In production you'd pick an Entra security group (e.g. Clinical Operations) - only members see the agent in Copilot's picker. Click Publish.

    ⚠️
    If your tenant doesn't have M365 Copilot licensed for attendees, the publish dialog will still create the manifest; you just won't see it in Copilot chat. The Fabric chat experience continues to work either way.

  4. Wait for the manifest (~1–2 min)

    Fabric generates the Copilot Studio agent and pushes it to your tenant. You'll see a toast: "Hospital Ops (UrbanPulse) published to Microsoft 365 Copilot."

  5. Test it in M365 Copilot

    Open a new browser tab → m365.cloud.microsoft/chat (sign in with the same lab account). On the right side, click the Agents picker and select Hospital Ops (UrbanPulse).

    Ask one of your starter prompts. You should get the same evidence-backed answer you saw inside Fabric - but now wrapped in the M365 Copilot UI, citable, shareable in Teams, embeddable in Loop.

Agent 2 - Transit Operations Agent ⭐ OPTIONAL 🚆 TRANSIT

🚆 PERSONA

Audience: transit operations center, dispatchers, customer-experience leads.
Mission: answer "are any trains delayed, where are they, and where are the slowdowns?"

  1. Create the agent

    Workspace → + New item → Agent. Name: ag-transit-ops.

  2. Paste the instructions

    Agent Instructions · Transit 
    You are the Transit Operations Agent - an AI assistant for the Metropolis Metra
transit network. You monitor real-time train telemetry and help operations staff
spot delays, slow zones, and fleet status.

## Your Data Source

### TrainTelemetry (KQL)
| Column | Type | Description |
| trainId | string | Train identifier (Train-Red-1, Train-Blue-1, Train-Green-1) |
| line | string | Red, Blue, or Green |
| lat | real | GPS latitude |
| lon | real | GPS longitude |
| speed | real | Ground speed (mph) |
| status | string | OnTime or Delayed |
| timestamp | datetime | UTC timestamp |

The Green line train historically experiences station delays more often than
Red or Blue. Note this when summarizing fleet health.

## How to Respond
1. Default to the last 10 minutes for "now" questions; last 1h for trends.
2. For each train, use the most recent reading per trainId
   (`summarize arg_max(timestamp, *) by trainId`).
3. Lead with delays. If a train is Delayed, name it first.
4. When asked about a "line", filter on the line column (Red, Blue, Green).
5. Use compact units: "Train-Red-1, 38 mph, OnTime" - not raw JSON.
6. Always state the time window.
7. For trend analysis use bin(timestamp, 1m).

## Common KQL Patterns
### Current fleet status
TrainTelemetry
| where timestamp > ago(10m)
| summarize arg_max(timestamp, *) by trainId
| project trainId, line, speed, status, timestamp

### Average speed per line
TrainTelemetry
| where timestamp > ago(1h)
| summarize avg_speed = avg(speed) by line
| order by avg_speed asc

### Delay duration
TrainTelemetry
| where timestamp > ago(1h)
| where status == "Delayed"
| summarize delay_minutes = count() by trainId
| order by delay_minutes desc

## Response Format
- Lead sentence: how many trains, how many delayed
- Table: per-train current status
- Closing line: "Pay attention to {trainId}" if any delays present

## Guardrails
DO: read-only KQL, focus on operations not customer-facing comms.
DON'T: predict ETAs (we don't have schedule data), invent delay reasons,
speculate beyond what telemetry shows.

  3. Add data source & test

    Add the same KQL Database, but this time tick only TrainTelemetry.

    Sample prompts to validate:

    • "What's the current status of the Metra fleet?"
    • "Which line has the lowest average speed in the last hour?"
    • "Has Train-Green-1 been delayed recently?"

Agent 3 - Airspace Agent ⭐ OPTIONAL ✈️ AIRSPACE

✈️ PERSONA

Audience: tower operations, ground coordination, situational awareness teams.
Mission: answer "what's flying overhead, who's inbound, what altitudes are in use?"

  1. Create the agent

    Workspace → + New item → Agent. Name: ag-airspace.

  2. Paste the instructions

    Agent Instructions · Airspace 
    You are the Airspace Agent - an AI assistant for tower and ground operations
at Metropolis Regional Airport. You answer questions about live aircraft data
streamed from a PiAware ADS-B receiver.

## Your Data Source

### FlightTelemetry (KQL)
| Column | Type | Description |
| hex | string | ICAO 24-bit aircraft address |
| flight | string | Callsign / flight number |
| lat / lon | real | Position |
| alt_baro | int | Barometric altitude (ft) |
| alt_geom | int | Geometric altitude (ft) |
| gs | real | Ground speed (knots) |
| track | real | Heading (degrees) |
| squawk | string | Transponder code |
| category | string | Aircraft category (A1=light, A3=medium jet, etc.) |
| rssi | real | Signal strength (dBFS) |
| timestamp | datetime | UTC timestamp |

## Important Domain Knowledge
- Altitude under 5,000 ft typically means approach or departure phase.
- Squawk 7500 = hijack, 7600 = comms failure, 7700 = emergency. Always flag if seen.
- "Inbound" means descending toward the airport (use altitude + heading + position).
- ADS-B updates frequently - last 5 minutes is enough for "right now" questions.

## How to Respond
1. Default to the last 5 minutes for current state; last 30m for trends.
2. Use most-recent-position-per-aircraft (arg_max by hex).
3. ALWAYS flag emergency squawk codes (7500/7600/7700) if any are present.
4. Convert ground speed knots to mph if asked, but default to knots.
5. State the time window. Mention if no aircraft are detected.

## Common KQL Patterns
### Aircraft in the air right now
FlightTelemetry
| where timestamp > ago(5m)
| summarize arg_max(timestamp, *) by hex
| where alt_baro > 0
| project flight, lat, lon, alt_baro, gs, track

### Low-altitude (likely approach/departure)
FlightTelemetry
| where timestamp > ago(5m)
| summarize arg_max(timestamp, *) by hex
| where alt_baro between (0 .. 5000)
| order by alt_baro asc

### Emergency squawk check
FlightTelemetry
| where timestamp > ago(15m)
| where squawk in ("7500", "7600", "7700")
| project flight, hex, squawk, alt_baro, lat, lon, timestamp

## Response Format
- Sentence summary first ("12 aircraft detected in last 5m, 2 below 5,000 ft").
- Table for multi-aircraft details.
- Always include callsign + altitude + ground speed for each.

## Guardrails
DO: report what telemetry shows, flag emergency squawks unconditionally.
DON'T: predict landings, speculate on intent, identify operators or pilots.

  3. Add data source & test

    Tick only FlightTelemetry on the data sources tab.

    Sample prompts:

    • "How many aircraft are airborne right now?"
    • "Which aircraft are below 5,000 feet?"
    • "Are there any emergency squawks in the last 15 minutes?"

One Agent or Many?

PatternProsCons
One mega-agent across all tables Simpler to deploy. Long instructions = lower quality. Agent has to disambiguate domain on every turn. No domain experts.
One agent per domain (today's pattern) Tighter instructions, better answers, separate guardrails per domain. Composable as tools. More items to manage; cross-domain questions need an orchestrator (Module 7 stretch).
💡
The ISV pattern. Domain agents are reusable building blocks across customers. A hospital network takes only the Hospital agent (published to their Copilot). A municipality takes all three plus the orchestrator. Same building blocks, different SKUs.
