Module 4 · Curate Phase

Transform Your Ingested Data for Analytics

Modules 2 and 3 landed three classes of data into Fabric - mirrored operational tables, an uploaded facility catalog, and live KQL streams. None of it is yet shaped for the dashboards, semantic models, and agents you'll build in the next four modules.

Fabric gives you several ways to shape data once it has landed. In this module you'll use two of them together: a notebook to clean and join the operational tables into a curated Silver layer, and SQL views on the Lakehouse SQL endpoint to expose Gold-style aggregates for Power BI and the agents downstream. Dataflow Gen2 and Materialized Lake Views are noted as no-code and managed alternatives.

⏱ 40 minutes
🎯 Goal: Bronze → Silver → Gold curated tables
🛠 Notebook (Spark SQL) · SQL views · Lakehouse

Learning Objectives

  • Inspect what landed in lh_urbanpulse_bronze from M2 (mirrors + uploaded catalog).
  • Run a Fabric notebook (Spark SQL) to join hospitals, wards, and regions into a curated silver_hospital_facility Delta table.
  • Create SQL views on the Lakehouse SQL analytics endpoint for downstream BI and agents.
  • Recognize when to reach for Dataflow Gen2 (no-code) or Materialized Lake Views (managed refresh) instead.

Why a Transform Step?

What landed in M2 is operational shape: each mirrored table looks like the source system, and the uploaded facility catalog is denormalized for storage, not analysis. That's fine for an audit trail, but it's not what a Power BI report or a Data Agent should see. Operators ask questions like "how many beds are available in Region North?" - which spans Hospitals, Wards, and Regions, and requires consistent column names and types.

Transforming the Bronze layer into a curated Silver layer (clean, joined, typed) plus a Gold layer of BI-ready aggregates is what makes the rest of the lab possible. The medallion split also lets you evolve raw schemas without breaking downstream consumers.

Approach Best for You'll use it in
Notebook (Spark Python / Spark SQL) Joins, type cleanup, derived columns, ML prep, complex logic Step 2 (Silver build)
SQL view on the SQL endpoint Lightweight aggregates and friendly names for BI / agents Step 3 (Gold views)
Materialized Lake View SQL transforms you want persisted as Delta with managed refresh Step 4 (optional)
Dataflow Gen2 (Power Query) Visual, no-code prep - filtering, type changes, joins Reference only
ⓘ
Pick the approach that fits your scenario. Fabric does not force one path - this module deliberately chooses notebook + SQL view because they are the two patterns you will see most often in real ISV deployments. The references at the bottom of this page point to the official docs for every other option.

Step 1 - Inspect What Landed in Bronze LAKEHOUSE

  1. Open the Bronze Lakehouse

    From your workspace, open lh_urbanpulse_bronze.

    In the left-hand explorer you should see the tables you created in M2:

    • Hospitals, Wards, Staff — mirrored from Azure SQL.
    • regions, trainRoutes — mirrored from Cosmos DB for NoSQL.
    • facility_catalog — uploaded Parquet (or shortcut, depending on your M2 path).

    Click any table and use the Preview pane to confirm rows are present.

  2. Switch to the SQL analytics endpoint

    In the upper-right corner of the Lakehouse, switch from Lakehouse view to SQL analytics endpoint view. This is the same Delta data, surfaced as a T-SQL endpoint for read-only queries and views.

    Run a quick sanity query:

    SELECT TOP 5 * FROM Hospitals;
SELECT TOP 5 * FROM Wards;
SELECT TOP 5 * FROM regions;

Step 2 - Build a Silver Table with a Notebook NOTEBOOK

Now you'll create a curated silver_hospital_facility Delta table that joins Hospitals, Wards, and regions into a single analytics-friendly shape. A Spark notebook is the right tool because the join spans tables originating from two different mirrors (Azure SQL + Cosmos DB) and we want consistent typing on the way out.

  1. Create a new notebook bound to the Bronze Lakehouse

    From the workspace, click + New item → Notebook.

    Name it nb_urbanpulse_silver.

    In the left explorer of the notebook, click + Lakehouse and pick lh_urbanpulse_bronze as the default Lakehouse. This makes spark.sql("SELECT * FROM Hospitals") resolve without any extra path.

  2. Write the Silver transform (Spark SQL)

    Paste the following into the first cell. This produces one wide row per ward, with the hospital and region attributes joined in:

    silver_df = spark.sql("""
    SELECT
        w.WardId             AS ward_id,
        w.Name               AS ward_name,
        w.Beds               AS beds,
        h.HospitalId         AS hospital_id,
        h.Name               AS hospital_name,
        h.Address            AS hospital_address,
        h.City               AS hospital_city,
        r.regionId           AS region_id,
        r.regionName         AS region_name,
        CURRENT_TIMESTAMP()  AS curated_at
    FROM Wards     w
    JOIN Hospitals h ON h.HospitalId = w.HospitalId
    LEFT JOIN regions r ON r.regionId = h.RegionId
""")

silver_df.write \
    .format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .saveAsTable("silver_hospital_facility")

    Run the cell. The output should report a successful Delta write with a row count.

  3. Validate the Silver table

    Add a second cell:

    display(spark.sql("""
    SELECT region_name, COUNT(*) AS wards, SUM(beds) AS total_beds
    FROM silver_hospital_facility
    GROUP BY region_name
    ORDER BY total_beds DESC
"""))

    You should see one row per region with the total number of wards and beds. This is the shape every downstream consumer (BI report, ontology, hospital agent) will read from.

💡
Why save as a managed Delta table? Writing with saveAsTable registers the table in the Lakehouse metastore so it shows up automatically in the SQL endpoint, in Power BI's Direct Lake mode, and in Fabric Data Agents. No extra plumbing.

Step 3 - Create Gold SQL Views SQL ENDPOINT

Gold-tier objects in this lab are intentionally lightweight: SQL views on the Lakehouse SQL analytics endpoint. They give Power BI and the agents you'll build later a stable, friendly contract without copying any data.

  1. Open the SQL analytics endpoint of the Bronze Lakehouse

    From the workspace, open lh_urbanpulse_bronze and switch the upper-right toggle to SQL analytics endpoint.

    Click New SQL query.

  2. Create a region-level capacity view

    Paste and run:

    CREATE OR ALTER VIEW gold_region_capacity AS
SELECT
    region_id,
    region_name,
    COUNT(DISTINCT hospital_id) AS hospitals,
    COUNT(*)                    AS wards,
    SUM(beds)                   AS total_beds
FROM silver_hospital_facility
GROUP BY region_id, region_name;

  3. Create a transit lines summary view

    The transit topology lives in trainRoutes from the Cosmos mirror. Add a lightweight Gold view for it too:

    CREATE OR ALTER VIEW gold_transit_lines AS
SELECT
    line,
    COUNT(*) AS station_count
FROM trainRoutes
GROUP BY line;

  4. Sanity-check both views

    SELECT * FROM gold_region_capacity ORDER BY total_beds DESC;
SELECT * FROM gold_transit_lines  ORDER BY line;

    Both should return rows. These are the views your Power BI report (M6) and Hospital agent (M8) will consume.

Step 4 - Materialized Lake View OPTIONAL

If you want the region capacity numbers persisted as Delta with Fabric-managed refresh (instead of computed on every query), promote the Gold view to a Materialized Lake View. This is a preview-tier feature; skip it if your workspace doesn't have it yet.

  1. Create the materialized view

    From the Bronze Lakehouse SQL endpoint, run:

    CREATE MATERIALIZED LAKE VIEW mlv_region_capacity AS
SELECT
    region_id,
    region_name,
    COUNT(DISTINCT hospital_id) AS hospitals,
    COUNT(*)                    AS wards,
    SUM(beds)                   AS total_beds
FROM silver_hospital_facility
GROUP BY region_id, region_name;

    Fabric will manage the refresh as the underlying Silver table changes. This is useful when the aggregates are read very frequently and recomputation cost matters.

Alternative Transformation Paths

You picked notebook + SQL view in this module because that's the most common ISV pattern. Three other paths cover different team preferences:

  • Dataflow Gen2 (Power Query): visual, no-code data prep. Best for analysts who already know Power Query and want to land curated tables without writing Spark or T-SQL.
    Tutorial - Copy & transform with Dataflow
  • Pipelines / Data Factory: orchestrate notebooks, dataflows, and copy activities on a schedule. Layer this on top of any of the above when you need automation.
  • T-SQL in a Warehouse: if your downstream pattern is Warehouse-first (multi-table writes, transactions), do the same shaping with T-SQL stored procedures inside a Fabric Warehouse rather than views on the Lakehouse SQL endpoint.

Validation

  • silver_hospital_facility exists as a managed Delta table in the Bronze Lakehouse.
  • gold_region_capacity and gold_transit_lines views return rows from the SQL endpoint.
  • You can name at least one scenario where Dataflow Gen2 or a Materialized Lake View would be the better choice.

What You Just Built

  • A curated Silver Delta table joining hospitals, wards, and regions across two source systems.
  • Two Gold SQL views ready for Power BI and the Data Agents.
  • (Optional) a Materialized Lake View for managed-refresh aggregates.
