Module 2 · Build Phase 1

Lakehouse, Shortcuts & Mirroring

Land three different data shapes - Parquet files, an operational SQL database, and a NoSQL Cosmos collection - into a single Bronze Lakehouse using three different ingest techniques. By the end of this module the warm side of your medallion is alive.

⏱ 45 minutes
🎯 Goal: 3 ingest patterns, 1 Lakehouse
🛠 Lakehouse · Mirror · Shortcut

Learning Objectives

  • Upload Parquet + image files from the lab repo into the Lakehouse Files zone.
  • Convert raw Parquet into managed Delta tables with one click.
  • Mirror an Azure SQL database into Fabric - zero-ETL, near-real-time.
  • Create a OneLake shortcut to a shared Cosmos DB JSON dataset.
  • Run a single SQL query that joins all three sources via the Lakehouse SQL endpoint.

Why Three Ingest Patterns?

Real ISV deployments hit data in many shapes. Fabric's value is that you don't have to pick one tool - you pick the right pattern per source and they all converge on OneLake.

Figure 2.1 - Three ingest patterns, one Lakehouse, one consumer surface. The "OneLake" promise.

Step 1 - Upload Parquet to the Lakehouse UPLOAD

  1. Open lh_urbanpulse_bronze

    From your workspace, click the Lakehouse item. The Explorer pane opens with two folders: Tables (empty) and Files (empty).

  2. Upload the tumor Parquet folder

    Right-click Files → Upload → Upload folder. The coach team has placed the parquet bundle in a SharePoint link shown on your seat slip - download the parquet/ folder to your local Downloads, then upload it here.

    This uploads the tumor metadata Parquet plus the supporting image files (~80 MB total).

    📷
    SCREENSHOT NEEDED
    module-2/01-files-upload.png
    Lakehouse Files explorer with the parquet/ folder uploaded and expanded

  3. Promote the Parquet to a Delta table

    Navigate into Files / parquet / Tumor Data. Right-click the .parquet file → Load to Tables → New table.

    Name it tumor_metadata and click Load.

    ⓘ
    Behind the scenes Fabric runs a Spark job that reads the Parquet, applies the schema, and writes a managed Delta table to Tables/tumor_metadata. Takes ~30 seconds.

  4. Preview the table

    Refresh the Tables list. Click tumor_metadata. You should see ~50 rows with columns: id, image_path, caption, modality.

Step 2 - Mirror an Azure SQL Database MIRROR

Mirroring brings your operational SQL data into Fabric continuously, with no ETL pipelines to author. Fabric reads the source's CDC log and writes Delta into OneLake.

  1. Create a Mirrored Database

    From the workspace, click + New item → search "Mirrored". Choose Mirrored Azure SQL Database.

    Name it: mir_urbanpulse_studentdb.

  2. Connect to the lab SQL DB

    In the connection wizard, paste the values from Appendix A · Lab Credentials:

Database: StudentDB

Username: lab_reader
Password: (see Appendix A · Lab Credentials)

  3. Choose tables to mirror

    Tick all three: Students, Classes, Enrollments. Click Mirror database.

  4. Watch the initial snapshot

    The mirror creates a snapshot first, then switches to incremental CDC. Status pane should reach Replicated in under two minutes.

    ✓
    Why this matters for ISVs: When you onboard a new customer, mirroring lets you pull their operational SQL data without deploying anything to their environment. No agents, no scheduled exports, no broken pipelines.

Step 3 - Shortcut a Cosmos DB Container SHORTCUT

Shortcuts are virtual references to data that lives elsewhere - no copy, no movement. They appear as tables/folders in your Lakehouse but query through to the source.

  1. Open your Lakehouse

    Back in lh_urbanpulse_bronze, right-click Tables → New shortcut.

  2. Choose the source

    Pick Azure Cosmos DB for NoSQL. Paste the read-only endpoint and key from Appendix A · Lab Credentials (rows Cosmos endpoint and Cosmos key).

    Browse to: StudentInfo database → Grades container.

  3. Configure the shortcut

    Name: cosmos_grades. Click Create.

    !
    Performance note. Shortcuts query the source on read. For Cosmos DB this means RU consumption on the source account. For chatty workloads you may want to materialize via a notebook into a real Delta table - covered in advanced labs.

Step 4 - One SQL Query Across All Three Sources

The payoff. Three different ingest mechanisms, but to a downstream consumer they're all just tables in your Lakehouse SQL endpoint.

  1. Switch to the SQL analytics endpoint

    Top-right of the Lakehouse, click the dropdown that says Lakehouse and choose SQL analytics endpoint.

  2. Run the unified query

    Click New SQL query and paste:

    T-SQL 
    -- A purposely-silly query that proves the three sources unify
SELECT
    s.StudentName,
    c.ClassName,
    g.Grade,
    t.modality   AS sample_imaging_modality,
    t.caption    AS sample_caption
FROM dbo.Students     AS s        -- mirrored from Azure SQL
JOIN dbo.Enrollments  AS e ON e.StudentId = s.StudentId
JOIN dbo.Classes      AS c ON c.ClassId   = e.ClassId
JOIN cosmos_grades    AS g ON g.Student_ID = s.StudentId   -- shortcut to Cosmos
CROSS APPLY (
    SELECT TOP 1 modality, caption
    FROM dbo.tumor_metadata                                -- uploaded Parquet → Delta
    ORDER BY id
) AS t
WHERE s.StudentId <= 5;

  3. Inspect the result

    You should see five rows, one per student, with grade from Cosmos and a sample row from the Parquet table. That's three different physical data stores in one query.

Validation

  • Lakehouse Tables shows tumor_metadata (uploaded), Students/Classes/Enrollments (mirrored), and cosmos_grades (shortcut).
  • The unified query above returns 5 rows.
  • OneLake explorer shows all four under one workspace, one Lakehouse.
