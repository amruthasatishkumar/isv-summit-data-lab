KQL Analytics & Real-Time Dashboard

Now that live data is flowing, write KQL queries across all three streams and pin them as live tiles to a Real-Time Intelligence Dashboard. By the end you'll watch the city pulse update on screen, second by second.

⏱ 60 minutes
🎯 Goal: live RTI dashboard
🛠 KQL Queryset · RTI Dashboard

Learning Objectives

  • Write KQL aggregation, summarize, and arg_max patterns over live data.
  • Use join across the Hospital Vitals + Movement tables.
  • Build a Real-Time Dashboard with five tiles spanning the hospital and transit domains.
  • Add parameters (time window, line filter) to make tiles interactive.
  • Configure auto-refresh and watch the dashboard update live.
  • (Optional stretch) Build a Bronze → Silver → Gold medallion inside KQL using update policies.

Step 1 - Build a KQL Queryset

A Queryset is a Fabric item that saves named KQL queries against a database. We'll build five queries here - they'll become our five dashboard tiles.

  1. Create the queryset

    Workspace → + New item → KQL Queryset. Name: qs_urbanpulse_ops.

    When prompted, attach it to your KQL Database (inside eh_urbanpulse_rti).

Step 2 - The Five Tile Queries

Run each query block. As soon as it returns rows, click Pin to dashboard at the top-right of the result pane (we'll create the dashboard in Step 3).

ⓘ
Default time window for "now" queries: last 15 minutes. For trends: last 1 hour. The KQL ago() function makes both trivial.

Tile 1 - Critical patient count SCALAR

KQL · Tile 1 
// Count of patients with critical vitals in the last 15 minutes
HospitalVitals
| where timestamp > ago(15m)
| summarize arg_max(timestamp, *) by patient_id
| where condition == "critical"
    or spo2 < 90
    or heart_rate > 120
    or temperature_f > 101.5
| count

Tile 2 - Top 5 highest heart rate TABLE

KQL · Tile 2 
// Most recent reading per patient, sorted by HR desc
HospitalVitals
| where timestamp > ago(15m)
| summarize arg_max(timestamp, *) by patient_id
| project patient_id, condition, heart_rate, spo2, temperature_f, timestamp
| order by heart_rate desc
| take 5

Tile 3 - Room occupancy bar chart BAR CHART

KQL · Tile 3 
// Live census per location - anyone whose latest event isn't 'Discharged'
HospitalMovement
| summarize arg_max(timestamp, *) by patient_id
| where event_type != "Discharged"
| summarize patients = count() by to_location
| order by patients desc
| render columnchart with (title="Live room occupancy")

Tile 4 - SpO2 trend for critical patients TIME SERIES

KQL · Tile 4 
// Per-minute SpO2 trend for any patient currently flagged critical
let critical_patients =
    HospitalVitals
    | where timestamp > ago(15m)
    | summarize arg_max(timestamp, *) by patient_id
    | where condition == "critical"
    | project patient_id;
HospitalVitals
| where timestamp > ago(30m)
| where patient_id in (critical_patients)
| summarize avg_spo2 = avg(spo2) by bin(timestamp, 1m), patient_id
| render timechart with (title="SpO2 trend · critical patients (last 30m)")

Tile 5 - Train delays by line PIE

KQL · Tile 5 
// Latest status for each train, grouped by line and on-time/delayed
TrainTelemetry
| where timestamp > ago(10m)
| summarize arg_max(timestamp, *) by trainId
| summarize trains = count() by line, status
| render piechart with (title="Train fleet status · last 10m")

Step 4 - Add Dashboard Parameters

Hardcoding 15-minute windows is OK for a demo, but real ISV dashboards need user-controlled filters.

  1. Add a "Time range" parameter

    Dashboard → Manage parameters → + Add:

    Parameter 
    Display name: Time range
Variable name: _timeRange
Type: Duration
Default: 15m
Values: 5m, 15m, 30m, 1h, 6h, 24h

  2. Wire it into Tiles 1, 2, 4

    Edit each tile's KQL - replace ago(15m) with ago(_timeRange). Save.

    The dashboard now has a dropdown at the top. Try switching from 15m → 1h → 24h and watch all three tiles re-execute against new windows.

  3. Add a "Train line" parameter

    Add a parameter _line of type String with values from the line column. Wire it into Tile 5:

    KQL · parameterized 
    TrainTelemetry
| where timestamp > ago(_timeRange)
| where _line == "All" or line == _line
| summarize arg_max(timestamp, *) by trainId
| summarize trains = count() by line, status
| render piechart

Step 5 - Medallion Inside KQL with Update Policies ★ OPTIONAL

⚠
This step is optional. The dashboard you built in Steps 1–4 is a complete deliverable. Add this section if you want to see how to build a Bronze → Silver → Gold medallion entirely inside KQL - no Spark notebooks, no scheduled pipelines. Allow ~15 extra minutes.

So far HospitalVitals, HospitalMovement, and TrainTelemetry are all Bronze - raw, schema-on-write copies of what landed via Eventstream. A KQL update policy is a stored expression that runs on every ingest into a source table and writes the transformed rows into a target table. Chain them and you get medallion tiers that stay in sync as new events stream in - no orchestrator needed.

Figure 4.2 - The KQL medallion: an update policy chains Bronze → Silver, a materialized view rolls Silver → Gold.

  1. 5a - Create the Silver target table

    In your KQL query window (Eventhouse → KQL Database → Query), run:

    KQL 
    // SILVER target - one clean, typed row per vitals event
.create table HospitalVitals_clean (
    patient_id: string,
    age: int,
    condition: string,
    heart_rate: int,
    bp_systolic: int,
    bp_diastolic: int,
    temperature_f: real,
    spo2: int,
    is_critical: bool,
    timestamp: datetime
)

.alter table HospitalVitals_clean policy streamingingestion enable

  2. 5b - Define the Bronze → Silver update policy

    An update policy needs two pieces: a function that transforms rows, and a policy declaration that wires it to a source table.

    KQL 
    // 1. The transform function (called by KQL on every batch ingest into Bronze)
.create-or-alter function HospitalVitals_to_clean() {
    HospitalVitals
    | where isnotempty(patient_id) and heart_rate > 0
    | extend is_critical = (
        heart_rate > 130 or heart_rate < 45
        or spo2 < 90
        or bp_systolic > 180 or bp_systolic < 80
      )
    | project patient_id, age, condition,
              heart_rate, bp_systolic, bp_diastolic,
              temperature_f, spo2, is_critical, timestamp
}

// 2. The policy: every ingest into HospitalVitals fires the function and writes to HospitalVitals_clean
.alter table HospitalVitals_clean policy update
```
[
  {
    "IsEnabled": true,
    "Source": "HospitalVitals",
    "Query": "HospitalVitals_to_clean()",
    "IsTransactional": true,
    "PropagateIngestionProperties": true
  }
]
```
    ⓘ
    IsTransactional: true means a Silver write only commits if the function succeeded - bad source rows don't break the chain. Combined with PropagateIngestionProperties, retention and caching policies inherit too.

  3. 5c - Define the Silver → Gold materialized view

    For Gold we don't need another physical table - a materialized view keeps an always-current "latest row per patient" projection that answers dashboard queries in milliseconds.

    KQL 
    .create materialized-view with (backfill = true) PatientStatus_now on table HospitalVitals_clean
{
    HospitalVitals_clean
    | summarize arg_max(timestamp, *) by patient_id
}

    backfill = true seeds the view from existing Silver rows (so it works immediately on the data already mirrored from Bronze). New Silver rows update the view incrementally.

  4. 5d - Verify the chain

    Run all three to see the tiers light up:

    KQL 
    HospitalVitals             | count    // BRONZE: total raw rows
HospitalVitals_clean       | count    // SILVER: typed + filtered, should match (no bad rows)
PatientStatus_now          | count    // GOLD: should equal active patient count (~15)

    Point a dashboard tile at PatientStatus_now and you've got a sub-second "current critical patients" KPI without a single ETL job.

  5. 5e (optional within optional) - Repeat for Trains

    The recipe is identical. For practice, try:

    • TrainTelemetry → TrainTelemetry_clean (drop GPS-zero rows; is_delayed = status == "DELAYED") → TrainStatus_now mat-view (arg_max by train_id)
💡
Why this matters for the rest of the lab. Module 6's Fabric Ontology can bind directly to HospitalVitals_clean instead of raw HospitalVitals - which means Module 7's Hospital Data Agent gets cleaned data with the is_critical flag pre-computed. Less prompt-shaping, more accurate answers. And because the OneLake-availability toggle from Module 3 cascades to every table, your Silver/Gold tiers are also available as Delta in OneLake automatically - so the Direct Lake semantic model can read them too.
