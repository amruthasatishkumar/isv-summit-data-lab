SQL Server Workaround

Use this path if you cannot use the shared UrbanPulse database for the Module 2 mirroring exercise. You stand up the same schema and data on your own Azure SQL Database, then mirror that into Fabric instead.

ⓘ
When to use this. The default Module 2 path mirrors the pre-provisioned shared UrbanPulse database. If you do not have access to the shared server, or you want a database you fully control, create your own Azure SQL Database and run the single setup script below against it.

Download the setup script

One SQL script that creates the schema, seeds the data (Hospitals, Wards, Staff), and enables Change Tracking so the database is ready to mirror into Fabric.

urbanpulse-sql-setup.sql · schema + seed + Change Tracking · idempotent

⬇ Download script (.sql)

Prerequisites

  • Your own Azure SQL Database on a logical SQL server in your subscription.
  • Permission to sign in to that server with your Azure (Microsoft Entra) credentials.
  • Workspace access in Fabric to create a mirrored database.

Step 1 · Turn on Managed Identity for the SQL server

Fabric Mirroring authenticates to your Azure SQL Database using a managed identity, so enable it before you mirror.

  1. In the Azure portal, open your SQL server (the logical server that hosts your database, not the database itself).
  2. In the left menu under Security, select Identity.
  3. Under System assigned managed identity, set Status to On and click Save.
  4. Confirm an object (principal) ID is shown once it saves.
ⓘ
Managed identity is set at the SQL server level, and it applies to the databases on that server.

Step 2 · Sign in to your SQL database with Azure credentials

  1. Open your SQL database in the Azure portal and select Query editor (preview), or connect from SSMS / Azure Data Studio.
  2. Choose Microsoft Entra authentication and sign in with your Azure account.
  3. If prompted about firewall access, add your client IP to the server firewall, then sign in again.

Step 3 · Run the setup script

  1. Open urbanpulse-sql-setup.sql from the download above.
  2. Paste the whole script into the query editor and run it top to bottom.
  3. The script creates the tables, seeds the data, and enables Change Tracking. The final query returns row counts so you can confirm the load.
✓
Expect Hospitals, Wards, and Staff in the result. The script is idempotent, so you can re-run it safely if a step fails partway.

Step 4 · Mirror your database into Fabric

  1. In your Fabric workspace, select New item and choose Mirrored Azure SQL Database.
  2. Point the connection at your own SQL server and database, and sign in with your Azure credentials.
  3. Select the Hospitals, Wards, and Staff tables to mirror.
  4. Start the mirror and wait for the initial snapshot to complete, then continue with Module 2 using your mirrored database in place of the shared one.
ⓘ
From here, every later module that reads the mirrored Hospitals / Wards / Staff tables works the same way, since the schema and data match the shared database.

Validation

  • System-assigned managed identity is On for your SQL server.
  • The setup script ran and returned row counts for all three tables.
  • Change Tracking is enabled on the database and on each table.
  • The mirrored database in Fabric shows the three tables with a completed initial snapshot.
