Module 6 · Semantic Layer

Build a Fabric Ontology over Your Data

Tables and columns describe how data is stored. An ontology describes what your data means - the entities (Patient, Train, Region), the relationships between them (a Train serves a Region; a Patient is admitted to a Hospital), and the hierarchies that flow through both. This module gives every downstream agent and Copilot experience a shared semantic vocabulary.

⏱ 25 minutes
🎯 Goal: a graph the agents can reason over
🛠 Fabric Ontology · OneLake

Learning Objectives

  • Understand what a Fabric Ontology is and when to add one to a workspace.
  • Create an Ontology item in your workspace.
  • Define entity types (Patient, Train, Region) and bind them to your Lakehouse + Eventhouse data.
  • Define relationships across domains so cross-domain questions become traversal, not joins.
  • Validate the ontology and preview an entity-centric query.

Why Add an Ontology?

Up to this point you've built a great data stack: a real-time Eventhouse, a Lakehouse, a Direct Lake semantic model. But every consumer - dashboards, reports, agents, Copilot - still has to know which table holds which fact and how to join them. That's a tax on every new question.

A Fabric Ontology lifts that tax. You define the conceptual world once - "a Patient has Vitals, is admitted to a Hospital, and lives in a Region. A Train runs on a Line and serves Regions" - and bind each entity to the tables that already exist in OneLake. Every Data Agent, every M365 Copilot prompt, and every Foundry orchestrator you build downstream then speaks the same vocabulary.

💡
The ISV angle. When you ship UrbanPulse to a second customer, the ontology is the part you reuse verbatim. The customer's tables can have different names, different schemas, even different storage - the ontology bindings change, but the conceptual model (and every agent grounded on it) stays identical. That's how you get from "consulting gig" to "product."

Step 1 - Create the Ontology Item

  1. Open your workspace

    From the Fabric portal, navigate back to urbanpulse-{your-id}. By now you should have: a Lakehouse, a Mirrored DB, an Eventhouse, a Real-Time Dashboard, and a Direct Lake semantic model.

  2. Create a new Ontology

    Click + New item → search "Ontology" → pick Ontology (preview).

    Name it: ont_urbanpulse → Create.

Step 2 - Define Entity Types & Bind to OneLake

An entity type is "the conceptual thing" plus a binding to where its data actually lives.

  1. Add the Patient entity

    In the Ontology editor, click + Add entity type. Configure:

    • Name: Patient
    • Display name: Patient
    • Source: Eventhouse eh_urbanpulse_rti → HospitalVitals
    • Identity property: patient_id
    • Other properties: age, gender, condition, diagnosis_code
    📷

  2. Add Train the same way

    Repeat the pattern with this binding:

    Entity Source Identity Properties
    Train Eventhouse → TrainTelemetry trainId line, status

  3. Add the cross-domain entities

    Two entities don't have a single dedicated source - they're logical and you'll bind them to small lookup tables in the Lakehouse:

    • Hospital → Lakehouse lh_urbanpulse_bronze → dim_hospital (one row today)
    • Region → Lakehouse dim_region (created by coaches; one row per Metropolis region)
    ✓
    If dim_hospital or dim_region isn't in your Lakehouse yet, ask a coach - we ship them as a tiny seed CSV. Or skip the cross-domain entities now and add them later; the rest of the ontology still works.

Validation

The Ontology editor's left rail should now show 4 entity types: Patient, Train, Hospital, Region. Click any one to see its properties and source binding.

Step 3 - Define Relationships

This is where an ontology earns its keep. You're declaring the meaningful connections between entities so a downstream agent can traverse them without having to know SQL joins.

  1. Add the same-domain relationships

    In the Ontology editor, click + Add relationship and create these three:

    From Name To Cardinality
    Patient admitted_to Hospital many-to-one
    Train on_line Line (or store as a string property if you skipped Line as an entity) many-to-one

  2. Add the cross-domain relationships

    These are the magic ones - they're what lets M8 ask cross-domain questions without bespoke SQL:

    From Name To Why it matters
    Hospital in_region Region "How's hospital load in this region?"
    Line serves_region Region "Are trains for this region delayed?"

Step 4 - Validate & Preview

  1. Click Validate

    The editor checks that every entity has a valid OneLake binding and every relationship resolves on both sides. Fix anything red. Common gotchas:

    • Identity property typo (case-sensitive)
    • Binding to a Lakehouse table that doesn't exist yet (dim_region)
    • Cardinality mismatch (one-to-many vs. many-to-one)

  2. Try an entity-centric query

    Open the Explore tab. Type a natural-language probe like:

    Ontology Explore 
    list 5 Patients with their admitted_to Hospital and the Hospital's in_region Region

    You should get a small entity-level result table - no SQL written by you. The ontology compiled the traversal into KQL/SQL behind the scenes.

Validation

Validation passes (no red errors) and the Explore query returns results that span at least two entity types. You're ready to ground a Data Agent on this ontology in Module 7.

What You Just Built

  • An Ontology item ont_urbanpulse in your workspace.
  • Four entity types - Patient, Train, Hospital, Region - bound to your Eventhouse + Lakehouse data.
  • Four relationships, two of which cross domain boundaries via the shared Region anchor.
  • A validated semantic graph that the Module 7 Data Agent (and the optional Module 8 multi-agent catalog and Module 9 orchestrator) will ground on.
💡
Why this changes Module 7. Without the ontology, your Hospital agent in M7 would need to be told "the table is HospitalVitals, the patient column is patient_id, condition is in column...". With the ontology, you just point the agent at ont_urbanpulse and say "you ground on Patient + Vitals." The agent's instructions get shorter, more accurate, and reusable across customers.
