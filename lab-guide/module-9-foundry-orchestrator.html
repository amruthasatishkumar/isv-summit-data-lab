Foundry Connected-Agent Orchestrator ⭐ OPTIONAL

Wrap the Fabric Data Agents you built in Modules 7–8 inside an Azure AI Foundry connected-agent orchestrator, then publish that orchestrator to Microsoft 365 Copilot. Zero orchestration code - Foundry's connected-agents pattern handles specialist routing automatically.

⏱ 30 minutes (optional)
🎯 Goal: cross-domain Copilot agent
🛠 Azure AI Foundry · M365 Copilot
⭐
This module is optional. The lab is complete after Module 7 - you've already shipped a published Copilot agent. This module is the "make it the front door for cross-domain reasoning" stretch. Strongest if you built the full three-agent catalog in Module 8, but works with just the Hospital agent from Module 7 (you'll connect fewer specialists). No notebook code, no Python - entirely portal-driven.

Learning Objectives

  • Use Azure AI Foundry's connected-agent pattern to orchestrate specialist agents without code.
  • Register your Fabric Data Agents from Modules 7–8 as Foundry tools.
  • Attach Azure AI Search as a knowledge source for RAG-grounded recommendations.
  • Test cross-domain reasoning in the Foundry Playground with full trace visibility.
  • Publish the orchestrator to Microsoft 365 Copilot as the cross-domain front door.

How It Works

Single domain agents are useful; multi-agent orchestration is what unlocks cross-domain reasoning. We'll do this without writing any orchestration code - Azure AI Foundry has a built-in connected-agents pattern where a main agent automatically delegates to specialist agents you register as tools.

The flow:

  1. Open the pre-provisioned Foundry project in your tenant.
  2. Create a City Ops Orchestrator agent (gpt-4o-mini).
  3. Attach your Fabric Data Agents from Modules 7–8 as connected agents.
  4. Attach Azure AI Search as a knowledge source for RAG.
  5. Test in the Foundry Playground.
  6. Publish to Microsoft 365 Copilot with one click.
Figure 9.1 - Foundry orchestrator with connected Fabric Data Agents and AI Search knowledge. The Foundry agent is then published to M365 Copilot.

Build the Orchestrator

  1. Open your Azure AI Foundry project

    Open a new browser tab → ai.azure.com. Sign in with your lab account.

    On the home screen pick the pre-provisioned project: aifprj-isvsummit. (Project name and URL are listed in Appendix A · Lab Credentials.)

    ⓘ
    Why Foundry? Foundry is Microsoft's agent factory - it gives you a hosted agent runtime, a connected-agents pattern (one agent delegating to specialists as tools), built-in tracing, and a one-click publish path to M365 Copilot, Teams, and the web. Zero orchestration code.

  2. Create the City Ops Orchestrator agent

    In the left rail, click Agents → + New agent. Fill in:

    • Name: City Ops Orchestrator
    • Deployment: gpt-4o-mini (already in the project)
    • Instructions (paste exactly):
    Agent Instructions 
    You are the City Ops Orchestrator - chief of staff for a Smart City operations team.

For any user question, decide which of your connected specialist agents to call:
- Hospital Ops Agent - patient vitals, room occupancy, movement
- Transit Ops Agent  - train status, speed, delays

You may call multiple specialists in the same turn when a question spans domains.
After gathering specialist signals, search your knowledge base (city operations
playbooks) for the matching response procedure. Then synthesize ONE concise answer with:

  SITUATION   - what each specialist reported (one bullet per domain)
  ACTIONS     - playbook-recommended steps, cited by section number
  CONFIDENCE  - HIGH if all relevant specialists responded; MEDIUM otherwise

Be terse and operational. Do not invent data the specialists did not return.

    Click Create.

  3. Attach your Fabric Data Agents as connected agents

    On the agent canvas, scroll to Tools & Knowledge → + Add tool → Connected agent.

    For each Fabric Data Agent you built in Modules 7–8, add one connected-agent entry:

    Tool nameDescription (what it does)Endpoint
    hospital_opsAnswers questions about patient vitals, room occupancy, and movement on the hospital floor. Use for any clinical or capacity question.Hospital agent endpoint (Appendix A)
    transit_opsAnswers questions about train fleet status, speeds, and delays across Red/Blue/Green lines.Transit agent endpoint (Appendix A)
    ✓
    The description field is critical - that's what the orchestrator's LLM reads to decide when to call this tool. Be specific about what each agent is good for.

    If you only built the Hospital agent in M6, attach just that one - the orchestrator works with any number of connected agents.

  4. Attach Azure AI Search as knowledge

    Same panel: + Add knowledge → Azure AI Search. Pick the pre-provisioned connection srch-cityops and the index city-ops-index. Accept the defaults for embedding model and hybrid search.

    This gives your orchestrator the city-ops playbooks for grounded ACTIONS sections.

  5. Test in the Foundry Playground

    Click Try in Playground. Paste the canonical cross-domain question:

    ❓
    "Region A has train delays AND rising hospital load. What should the city do?"

    Watch the trace pane on the right - you'll see the orchestrator decide to call hospital_ops and transit_ops in parallel, then issue an AI Search query, then synthesize.

    Expected output shape:

    Playground Output 
    SITUATION
- Hospital: 4 critical patients, SpO2 trending down on PAT-10003.
- Transit: Train-Green-1 delayed 8/10 readings; Green line 11 mph below baseline.

ACTIONS (Region A spike playbook §3.2)
1. Activate hospital staffing tier-2.
2. Reroute Green-line passengers to Blue line shuttle bus #14.
3. Open incident bridge with transit + ED leadership.

CONFIDENCE: HIGH (2/2 relevant specialists responded; playbook §3.2 matched).

    Try a few more - the orchestrator picks which specialists to call automatically:

    • "How's hospital load looking right now?" → Hospital only
    • "What's the operational picture across the city?" → both agents
    • "Are there any concurrent emergencies?" → both agents
    📷
    SCREENSHOT NEEDED
    module-9/01-foundry-playground.png
    Foundry Playground showing the orchestrator's response with the trace pane open, listing 2 connected-agent calls and 1 AI Search call.

  6. Publish to Microsoft 365 Copilot

    From the agent canvas, click Publish → Microsoft 365 Copilot.

    • Display name: City Ops Orchestrator (UrbanPulse)
    • Description: Ask cross-domain questions about UrbanPulse city operations - hospital and transit. Answers cite the city ops playbooks.
    • Audience: Just me (for the lab)
    • Starter prompts:
      • What's the operational picture across the city right now?
      • Region A has train delays AND rising hospital load. What should we do?
      • Are there any concurrent emergencies?

    Click Publish. After ~1–2 minutes, open m365.cloud.microsoft/chat, pick City Ops Orchestrator (UrbanPulse) from the agent picker, and ask a cross-domain question.

    💡
    The full loop. Streaming telemetry → Lakehouse + KQL → Fabric Data Agents → Foundry connected-agent orchestrator → published to M365 Copilot. Your end user never sees Fabric, Foundry, or any code - just a Copilot chat that knows their business.

Validation

  • City Ops Orchestrator agent exists in your Foundry project.
  • At least one Fabric Data Agent is wired in as a connected agent.
  • AI Search knowledge is attached.
  • Playground answers the canonical question in under ~30 seconds.
  • Orchestrator is published and visible in the M365 Copilot agent picker.
